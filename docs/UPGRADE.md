# Upgrade

`windows11-zombie` upgrades in place. The installer is idempotent —
re-running `install` over an existing tree converges to the new
desired state.

## In-place upgrade

```powershell
# 1. Back up first.
pwsh -File scripts/Install.ps1 backup

# 2. Pull the new revision.
cd C:\path\to\windows11-zombie
git fetch --tags
git checkout v0.5.0

# 3. Re-run the installer.
pwsh -File scripts/Install.ps1 install

# 4. Verify.
pwsh -File scripts/Install.ps1 verify
Restart-Service Windows11Zombie-Chat
```

The installer:

* Re-copies the payload tree (no-op for unchanged files).
* Re-applies ACLs (`Set-AiZombieAcl`).
* Reconfigures the service (`sc.exe config`).
* Re-registers the Scheduled Tasks.
* Re-creates the firewall rule.

User data (`secrets\env`, `state\conversations.db`,
`etc\policy.yaml`, `etc\skills.d\`) is left untouched.

## Rollback

If the new version misbehaves:

```powershell
git checkout <previous-tag>
pwsh -File scripts/Install.ps1 install
pwsh -File scripts/Install.ps1 restore -Path <latest-backup.zip>
Restart-Service Windows11Zombie-Chat
```

Or fully roll back to the pre-upgrade snapshot:

```powershell
pwsh -File scripts/Uninstall.ps1 -Archive -AssumeYes
# install the previous version, then:
pwsh -File scripts/Install.ps1 restore -Path C:\ProgramData\AiZombie-backups\windows11-zombie-<stamp>.zip
```

## Version skew matrix

The agent runtime (`payload/agent/*`) and the installer
(`scripts/*`) ship from the same repo and are tested together. We
do **not** support mixing versions across that boundary.

| Component | Skew tolerated | Notes |
| --- | --- | --- |
| Installer ↔ agent runtime | none | always upgrade together |
| `policy.yaml` schema | one major version | new optional keys are ignored by older agents |
| `conversations.db` | forward only | migrations are tracked via `PRAGMA user_version` |
| Audit log format | forward only | hash chain stays valid across upgrades |

## Breaking changes

Breaking changes bump the **major** version and are announced one
minor release in advance under `## Deprecated` in
[`CHANGELOG.md`](../CHANGELOG.md).

## Conventional Commits and release-please

From `0.5.0` onwards `CHANGELOG.md` is generated by `release-please`
from [Conventional Commits](https://www.conventionalcommits.org/).
PRs that change user-visible behaviour must use a `feat:`, `fix:`,
or `feat!:` prefix so the changelog renders correctly.
